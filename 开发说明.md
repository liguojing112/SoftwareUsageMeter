# SoftwareUsageMeter 开发说明

## 2026-04-25 现场交付补充

本次补充来自第二次交付后的环境复盘和现场操作保护调整。核心结论：雇主电脑缺少 Visual C++ 运行库是收费框无法触发/程序异常的重要环境原因之一；同时为了降低非技术用户乱点导致流程错乱，程序加入了更强的启动说明和导出前全屏等待遮罩。

### Visual C++ 运行库打包

- 现场问题表现：雇主电脑进入像素蛋糕导出框后，本程序未弹出收费框，随后程序可能卡死或无有效日志。
- 复盘原因：目标电脑缺少 Visual C++ Redistributable，导致部分原生依赖（如 pywin32 / psutil / Qt 相关 DLL 链）加载异常。
- 当前处理：`build.spec` 已把常见 VC++ 运行库 DLL 从 `System32` 收集进 PyInstaller 的 `binaries`。
- 当前包含：
  - `msvcp140.dll`
  - `msvcp140_1.dll`
  - `msvcp140_2.dll`
  - `vcruntime140.dll`
  - `vcruntime140_1.dll`
  - `concrt140.dll`
- 后续要求：交付必须使用最新 `build.bat` / `build.spec` 打包，不要手动复制旧 exe，也不要删除 VC++ DLL 收集逻辑。

### 启动说明提示框

- `main.py` 中的 `StartupHintDialog` 已改为启动阶段的强提示。
- 现场人员必须点击“知道了”后，程序才进入监控流程。
- 提示语明确要求按下面顺序操作：
  1. 点击像素蛋糕导出按钮
  2. 弹出像素蛋糕导出框
  3. 程序弹出收费框
  4. 引导客户付款
  5. 输入管理员密码解锁
  6. 再让客户导出照片
- 设计目的：减少雇主或顾客在程序未准备好时乱点、重复启动或误关窗口。

### 导出前全屏等待遮罩

- `main.py` 新增 `ExportWaitOverlay`，用于显示“请等待 5 秒钟”。
- 遮罩作用：
  - 阻止顾客连续点击导出按钮
  - 给导出框识别和收费框触发争取时间
  - 降低慢电脑上像素蛋糕先完成导出的概率
- 触发入口包含两层：
  - 快速鼠标热区监听：20ms 轮询鼠标，检测 PixCake 窗口右上角黄色导出按钮区域，鼠标按下命中、松开后立即显示等待遮罩
  - 监控线程兜底信号：`process_monitor.py` 识别到导出按钮点击后发出 `export_button_pre_clicked`
- 注意：等待遮罩不等同于收费框。等待遮罩负责“拖住现场操作”，正式收费仍由导出检测确认后显示。

### 本轮建议回归

1. 在未安装 Visual C++ Redistributable 的干净 Windows 电脑或虚拟机上运行最新 exe，确认不再因缺 DLL 异常退出。
2. 启动后确认必须点击“知道了”才能进入监控。
3. 未打开像素蛋糕时确认不会计时。
4. 打开像素蛋糕后确认开始计时。
5. 点击右上角黄色导出按钮后，确认快速出现“请等待 5 秒钟”遮罩。
6. 等待遮罩结束后，确认正常弹出收费框。
7. 管理员输入密码确认后，确认像素蛋糕可继续操作。
8. 检查 exe 同目录生成 `app.log`、`startup.log` / `crash.log`。

## 2026-04-24 稳定性交付更新

本次更新主要面向第二次远程交付暴露出的现场稳定性问题，以及交付前在另一台 Windows 电脑上的兼容性验证结果。

### 启动体验

- `main.py` 新增 `StartupHintDialog`，启动后延迟显示非阻塞提示窗，说明程序已启动、正在等待 `PixCake.exe`。
- 启动提示窗复用 `config.json` 中的 `wallpaper_path`，未配置壁纸时使用浅色渐变兜底。
- 继续使用软件 OpenGL 相关 Qt 设置，降低远程桌面、核显/独显切换环境下的渲染兼容风险。
- 启动早期继续写入 bootstrap 日志，优先写 exe 同目录 `app.log`，失败时写 `%TEMP%\SoftwareUsageMeter_bootstrap.log`。

### 计时周期

- `find_pid_by_name()` 现在只接受带有效可见主窗口的 `PixCake.exe` 实例，避免后台残留进程导致提前计时。
- 目标窗口需要满足可见、标题有效、尺寸合理等条件，最小化到 `-16000` 这类离屏窗口不会被当作可计时主窗口。
- `_on_process_stopped()` 改为结束本轮并清零，而不是暂停后等待下次继续累计。

### 导出与恢复

- `suspend_target_processes()` 已做幂等处理，避免同一进程族被重复 suspend 后只 resume 一次，导致像素蛋糕半挂起。
- 管理员确认收款后会调用 `restore_target_interaction()`，并延迟补一次恢复日志。
- 付款后的导出尾迹增加保护窗口，避免同一次实际导出 worker 出现时被误判为下一次导出，从而二次弹收费框。

### 本轮建议回归

1. 启动 exe 后确认友好提示窗显示并自动关闭。
2. 未打开像素蛋糕时确认不会开始计时。
3. 打开像素蛋糕主窗口后确认从 0 开始计时。
4. 关闭像素蛋糕后确认计时清零。
5. 重新打开像素蛋糕后确认重新从 0 开始。
6. 普通导出和“导出至本地”均能弹出收费框。
7. 管理员确认收款后像素蛋糕可继续操作，且不会二次弹收费框。
8. exe 同目录能生成 `app.log`。

## 1. 系统概述

SoftwareUsageMeter 是运行在 Windows 门店设备上的计时计费工具，用于监控“像素蛋糕”使用过程，并在导出前弹出收费窗口。

当前核心目标：

- 监控目标进程 `PixCake.exe`
- 自动开始 / 暂停计时
- 在导出流程前触发收费弹窗
- 自动识别导出张数
- 管理员确认收款后恢复导出

技术栈：

- Python 3.11
- PyQt5
- psutil
- pywin32
- Pillow
- Windows OCR
- PyInstaller

## 3. 模块职责

### 3.1 `main.py`

职责：

- 创建 `QApplication`
- 初始化配置、计时器、监控器、收费弹窗、托盘管理器
- 连接信号
- 管理收费流程状态
- 写启动摘要日志

当前已加入：

- 监控线程延后启动
- 程序启动环境日志
- 配置摘要日志
- 启动说明强提示 `StartupHintDialog`
- 导出前 5 秒全屏等待遮罩 `ExportWaitOverlay`
- 快速鼠标热区监听，用于在截图 / OCR 之前提前拦截导出按钮点击

### 3.2 `process_monitor.py`

职责：

- 监控目标进程是否运行
- 收集导出证据
- 判断是否触发收费
- 自动识别导出张数
- 写监控诊断日志

当前导出触发采用多证据并行策略，任一路命中即可触发：

| 证据类型 | 说明 |
|---------|------|
| `export_hwnd` | 导出相关窗口命中 |
| `export_worker_pid` | 检测到导出子进程 |
| `export_visual` | 识别到黄色导出按钮 |
| `export_button_clicked` | 检测到用户点击导出按钮 |
| `summary_probe` | 摘要区 OCR 命中导出张数 |
| `centered_dialog_scan` | 中央导出对话框扫描命中 |

OCR 主要负责“数张数”，不再强依赖 OCR 成功才允许弹窗。

### 3.3 `payment_overlay.py`

职责：

- 显示全屏收费弹窗
- 展示费用明细
- 展示双二维码
- 锁定导出操作直到管理员确认

当前支持：

- 微信收款码
- 支付宝收款码
- 壁纸背景
- 管理员密码确认收款

### 3.4 `admin_panel.py`

职责：

- 管理后台配置页
- 管理员密码验证页

当前支持配置：

- 计时单价
- 单张导出单价
- 目标进程名
- 导出关键字
- 微信收款码
- 支付宝收款码
- 壁纸
- 管理员密码

### 3.5 `tray_icon.py`

职责：

- 托盘图标
- 托盘菜单
- 状态页显示

当前状态页支持：

- 壁纸
- 半透明白色卡片
- 滚动条
- 计时与费用状态展示

### 3.6 `config_manager.py`

职责：

- 读取 / 保存 `config.json`
- 提供统一配置访问接口

## 4. 关键配置项

`config.json` 当前主要字段：

```json
{
  "rate": 1.0,
  "export_rate": 1.0,
  "default_export_count": 1,
  "admin_password": "sha256",
  "qr_code_path": "",
  "wechat_qr_code_path": "",
  "alipay_qr_code_path": "",
  "wallpaper_path": "",
  "process_name": "PixCake.exe",
  "export_window_keywords": ["导出", "Export"],
  "monitor_interval_ms": 2000
}
```

说明：

- `wechat_qr_code_path` / `alipay_qr_code_path`
  - 当前双码配置字段
  - 若为空，会回退到旧字段 `qr_code_path`
- `default_export_count`
  - OCR 识别失败时的兜底导出张数
- `monitor_interval_ms`
  - 基础监控周期

可选调试字段：

```json
{
  "debug_export_capture": true,
  "centered_dialog_scan": true
}
```

说明：

- `debug_export_capture`
  - 开启后保存调试截图包
- `centered_dialog_scan`
  - 控制中央导出对话框扫描兜底逻辑
  - 当前默认值为 `true`

## 5. 日志与排障

### 5.1 日志位置

- 源码运行：项目目录下的 `app.log`
- exe 运行：exe 同目录下的 `app.log`

### 5.2 当前新增的日志层次

#### 启动摘要

由 `main.py` 记录：

- `运行环境`
- `配置摘要`

可快速确认：

- 是否跑的是 exe
- 使用的 Python / 可执行文件路径
- 当前工作目录
- 壁纸和双二维码文件是否存在
- 当前费率、目标进程、关键字

#### 监控器初始化

由 `process_monitor.py` 记录：

- 目标进程名
- 导出关键字
- 监控周期
- 启动保护时间
- 视觉预热时间
- 摘要探测间隔
- 是否开启居中扫描
- 是否开启调试截图

#### 导出检测诊断

运行时每隔几秒节流记录一次，字段包括：

- `pid`
- `main_hwnd`
- `capture_hwnd`
- `capture_title`
- `capture_class`
- `export_hwnd`
- `export_worker_pid`
- `image_ok`
- `visual_candidate`
- `button_clicked`
- `summary_probe`
- `cached_count`
- `has_evidence`
- `startup_guard`
- `visual_warmup_skip`
- `dialog_mode`
- `centered_scan`
- `trigger_reason`

#### 确认触发日志

最终触发收费框时会记录：

- `确认导出窗口`
- `确认导出页预判`
- `确认导出子进程`

并携带 `trigger_reason`。

### 5.3 建议排障顺序

当现场再次出现“收费框未触发”时，按下面顺序看 `app.log`：

1. 是否出现 `运行环境`
2. 是否出现 `配置摘要`
3. 是否出现 `监控器初始化`
4. 是否出现 `检测到目标程序启动`
5. 是否出现 `导出检测诊断`
6. 是否出现：
   - `确认导出窗口`
   - `确认导出页预判`
   - `确认导出子进程`

如果第 5 步有日志但第 6 步没有，通常就能从诊断字段看出是哪条证据链没有命中。

## 7. 打包

### 7.1 推荐方式

```powershell
build.bat
```

### 7.2 手动方式

```powershell
pyinstaller build.spec --clean --noconfirm
```

输出：

```text
dist\SoftwareUsageMeter.exe
```

### 7.3 当前打包注意点

- 打包前请关闭旧的 `SoftwareUsageMeter.exe`
- `build.bat` 已加入旧进程清理逻辑
- exe 运行日志会写到 exe 同目录，便于现场收集
- `build.spec` 已显式收集 Visual C++ 运行库 DLL；后续打包时不要删除 `_vc_binaries` 相关逻辑
- 交付给雇主时只发最新 `dist` 产物，避免目标电脑继续运行旧 exe 导致“没有 VC++ 运行库”的问题复现
- 如果现场出现 exe 无法启动、状态页空白、收费框不弹且 `app.log` 无记录，优先怀疑环境依赖 / 安全软件 / 运行了旧副本，而不是先改检测逻辑
